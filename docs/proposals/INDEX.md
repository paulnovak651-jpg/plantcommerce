# Proposal Index

> **Label:** Proposed / review only

Use this folder for draft plans that agents should be able to read, review, and critique without treating them as approved work.

## Rules

- Proposal docs do not authorize implementation by themselves.
- Do not treat a proposal as current work unless Paul explicitly says to execute it or the work is promoted into the current root-doc flow.
- Proposal docs should say what problem they address, what is in scope, what is out of scope, and what open questions remain.
- Proposal docs should not claim to be the current task board.
- If a proposal conflicts with root docs, root docs win until the proposal is adopted.

## Recommended Label Block

Use this near the top of each proposal doc:

```md
> **Label:** Proposed / review only
> **Implementation status:** Not approved
> **Location status:** Safe to read, not safe to execute without explicit approval
```

## Promotion Rule

A proposal becomes executable only when one of these happens:

1. Paul explicitly says to implement it.
2. The work is promoted into `ROADMAP.md` or another explicitly current root doc.
3. A current task note references the proposal and states that implementation is approved.

## Suggested Contents

- goal
- why now
- proposed scope
- non-goals
- implementation sketch
- risks
- open questions
- acceptance criteria
